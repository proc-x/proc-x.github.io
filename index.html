Event-Driven Autonomous Work

目標は
「お客様の事務処理をなくす」こと

PROC.Xは業務のイベントポイントを正確に捉え、各イベントをトリガーとして 数十の専門AIエージェントが自律的に連携する基盤を構築します。 ただし、最終責任を曖昧にしないために、重要局面では必ず人間が介在する設計を採用します。

  • イベントポイント起点
  • 専門エージェント数十体

目指す運用状態

0 人が行う定型処理
~40 並行する専門エージェント
3 人間承認ゲート
24/7 業務イベント監視

人の作業を消しながらも、責任ある判断ポイントは人に残す。それが当社の設計原則です。

Step 1: 業務をイベント単位に分解する

まず手順書ではなく、業務の発生点を定義します。ここで作る分解図が、後続の実装と運用の土台になります。

1. Capture

受注、問い合わせ、申請、アラートなどの発生点を統一フォーマットで捕捉。

2. Route

重要度・期限・ドメインに応じて、最適な専門エージェント群へ即時ルーティング。

3. Execute

調査、判断、実行、報告を分業し、必要時のみ人間承認へエスカレーション。

4. Learn

完了ログを再学習し、次回イベントで判断速度と精度を継続改善。

Step 2: 仕事の流れを同じルールで設計する(やさしい圏論)

難しい数式は使いません。Step 1で作った分解図を「状態」と「次に進める処理」で表し、 誰が見ても同じ流れで判断できる設計ルールにそろえます。

点: いまの状態

例: 「問い合わせ受信済み」「回答案作成済み」「送信承認待ち」など。

矢印: 状態を進める処理

例: 分類する、下書きを作る、承認依頼を送る、顧客へ返信する。

つなぐ: 処理を連結する

「受信→分類→回答案→承認→送信」のように、手順を切れ目なく接続します。

問い合わせ対応を1本の流れで見ると

  1. 受信済みから開始
  2. 分類するで「担当確定」に進む
  3. 回答案を作るで「回答案完成」に進む
  4. 承認するで「送信可能」に進む
  5. 送信するで「対応完了」に到達

この見方の利点は、途中の状態と次の処理が明確になることです。 つまり「誰がどこを自動化し、どこだけ人が判断するか」を迷わず決められます。

Step 3: 専門AIエージェントを実装し、業務を自律実行する

単一AIではなく、役割が明確なマルチエージェント構成を採用します。必要な群だけ起動し、 実行後は監査ログを残して終了するため、運用時の追跡性と再現性を確保できます。

  1. Planner: イベント理解と実行計画の構築
  2. Analyst: 事実確認、規約照合、必要情報の抽出
  3. Executor: API連携、台帳更新、通知送信の実行
  4. Reviewer: 品質・リスクを検証し、再実行判定
  5. Auditor: 監査証跡を保存し、異常時に是正タスクを起票

代表的な専門エージェント群

見積/契約エージェント

契約条件の整合チェック、見積作成、承認依頼まで自律実行。

請求/入金エージェント

請求生成、入金突合、未収アラート、経理連携を自動化。

CS対応エージェント

問い合わせ分類、回答草案、対応履歴更新、再発防止連携を実施。

障害一次対応エージェント

ログ要約、原因仮説生成、一次切り分け、オンコール通知を担当。

購買/在庫エージェント

閾値監視から発注案作成、承認依頼、発注反映までを一貫処理。

ガバナンスエージェント

ポリシー違反、承認漏れ、個人情報リスクを常時監視。

Step 4: 人間承認ゲートを設計し、品質と責任を守る

人間介在は障害ではなく品質装置です。金額、契約、法務、顧客影響の大きい局面だけに承認を挿入し、 速度を落とさず事故リスクを下げます。

  • Gate A: 対外送信前レビュー(顧客影響の最終確認)
  • Gate B: 高リスク変更承認(契約/金額/法規制の担保)
  • Gate C: 例外復旧判断(自動復旧不可時の意思決定)
  • Post Review: 実行ログ監査とルール更新
完全自動化を目的化しません。責任境界を明確化した上で、 人が関与すべき瞬間だけを狙って残します。

Step 5: 90日で本番運用へ段階導入する

小さく始めて、実運用で例外を吸収しながら対象業務を広げます。段階導入で停止リスクを抑えます。

  • 承認ゲート初期設定

Day 31-60

  • 監査ログ整備

  • 部門横断のイベント連携
  • KPIダッシュボード運用開始

最終的に出来上がるものと、現場での効果

サイトを上から読むと、私たちの作業順に「設計」「実装」「運用」がつながって見える構成です。 最終的には次の3点が揃い、現場の速度と品質を同時に引き上げます。

1. 業務オーケストレーション設計書

イベント分解、状態遷移、承認条件、責任境界を明文化した運用設計。

役立つこと: 担当者が変わっても運用判断が揃う。

2. 自律実行するエージェント基盤

業務ごとに必要な専門エージェントが連携し、定型処理を自動実行。

役立つこと: 人の手作業と待ち時間が減り、処理が速くなる。

3. 承認ゲート付きの監査運用

重要判断は人が承認し、全実行ログを追跡できる状態を維持。

役立つこと: 品質と説明責任を保ったまま自動化を進められる。

よくある質問

本当に「人が何もしない」状態を目指せるのですか？

定型処理は限りなくゼロに近づけます。ただし責任判断はあえて人間ゲートを残す設計です。

エージェントが増えると管理が複雑になりませんか？

イベント駆動の起動制御と監査ログ設計により、常時起動を避けつつ責任追跡を可能にします。

既存SaaSや基幹システムを置き換える必要はありますか？

置き換えは不要です。既存システムを活かし、API・ファイル連携・通知連携で段階導入します。
